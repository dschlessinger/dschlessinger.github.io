Technical Blog: Week 6

Inheritance vs. Composition

August 24, 2014


Ruby's object-oriented nature caters to many different kinds of functionality. Two ways of structuring your code, inheritance vs. composition, will depend on the intended structure of your objects. For example, imagine you define a Person class:

class Person
  def walk
  	puts "I'm walking."
  end

  def run
  	puts "I'm running."

  def sit
  	puts "I'm sitting."
  end
end

Person.new.walk #=> "I'm walking."
Person.new.run #=> "I'm running."
Person.new.sit #=> "I'm sitting."

Since athletes are a subset of people, you can define an Athlete class and inherit the methods of the person class, while adding some methods that only athletes can do.

class Athlete < Person
  def sprint
  	puts "I'm sprinting."
  end
end

Athlete.new.walk #=> "I'm walking."
Athlete.new.run #=> "I'm running."
Athlete.new.sit #=> "I'm sitting."
Athlete.new.sprint #=> "I'm sprinting."

But if you tried this...

Person.new.sprint #=> undefined method `sprint'

...it wouldn't work. It's clear to see, then, that inheritance should be used when your objects follow a natural hierarchy.

Composition is different. When objects don't follow a natural hierarchy, use composition:

class Person
  def walk
  	puts "I'm walking."
  end

  def run
  	puts "I'm running."

  def sit
  	puts "I'm sitting."
  end
end

class Animal
  def growl
  	puts "Grrrrrrrr."
  end
end

Person.new.walk #=> "I'm walking."
Animal.new.growl #=> "Grrrrrrrr."
Person.new.growl #=> undefined method `growl'
Animal.new.walk #=> undefined method `walk'

Ruby has a lot of amazing functionality. Think carefully about your structure before writing your code. Depending on what you decide, one of these methods will probably suit your needs.
