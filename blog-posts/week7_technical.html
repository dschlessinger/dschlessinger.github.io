Technical Blog: Week 7

SQL vs. NoSQL

August 31, 2014


This blog post will be short, but sweet. There seem to be two main types of database architecture out there: SQL, which stands for Structured Query Language, and NOSQL, which stands for Not Only SQL.

SQL has been used for decades - it was invented in the early 1970s and is still used in nearly all database architecture. It is rigid but reliable, and everyone has used it so long that for the most part, all the kinks are worked out.

But just because it works doesn't mean it is the best. For example, querying massive databases can get to be very slow, and SQL isn't very D.R.Y. There are better ways to store data. Take, for example, a JSON - JavaScript Object Notation - which is sort of like a hash of a hash (h/t JSON Example):

{"menu": {
  "id": "file",
  "value": "File",
  "popup": {
    "menuitem": [
      {"value": "New", "onclick": "CreateNewDoc()"},
      {"value": "Open", "onclick": "OpenDoc()"},
      {"value": "Close", "onclick": "CloseDoc()"}
    ]
  }
}}

The content above really doesn't matter, but what does matter is that it's a very different from a table. SQL is relational, and NOSQL is non-relational - NOSQL data is often stored in key-value pairs like above. Therefore, SQL can only pull information that is arranged in tabular form, but NOSQL can pull information stored in many different forms, some like the JSON example above. NOSQL also works with wide column stores and graph databases. This flexibility cuts down on repeated information by catering to the specific needs of the program, which results in increased speed. SQL's speed, on the other hand, is limited by the code architecture. You could have the fastest servers in the world, but you'd still have to query all those tables.

It seems like more and more websites are finding NOSQL more fitting to their needs than SQL. I'm very excited to work with it in the future.
